# DrayTek / site2000.com 入口關係盤點 v1

## 目的

釐清：
- `site2000.com` / `www.site2000.com` 目前指到哪裡
- 80 / 443 / 5001 分別由誰處理
- 為什麼主站 HTTPS 與 DSM 管理入口會出現不同憑證與行為

---

## 一、DNS 解析

目前查到：
- `site2000.com` → `220.135.108.47`
- `www.site2000.com` → `220.135.108.47`

結論：
- 主網域與 `www` 都指向同一個外部 IP
- 網域入口沒有在 DNS 層分開

---

## 二、HTTP / HTTPS / DSM 的實際行為

### `http://site2000.com`
- 回應：`200 OK`
- Server：`nginx`
- 類型：看起來是網站入口

### `http://www.site2000.com`
- 回應：`200 OK`
- Server：`nginx`
- 類型：公開網站入口

### `https://site2000.com`
- 回應：`302 Found`
- 位置：`/weblogin.htm`
- 結論：HTTPS 主站目前不是正常網站，而是被導到路由器登入頁

### `https://www.site2000.com`
- 回應：`302 Found`
- 位置：`/weblogin.htm`
- 結論：HTTPS `www` 也被導到路由器登入頁

### `https://www.site2000.com:5001`
- 回應：`200 OK`
- 類型：Synology DSM
- 憑證：已改為 Let’s Encrypt，對 `www.site2000.com` 有效

---

## 三、真正的入口分工

### Port 80
- 目前由 **nginx 網站** 提供內容
- 看起來是正常網站入口

### Port 443
- 目前由 **DrayTek 路由器** 接手
- 會把請求導到 `/weblogin.htm`
- 憑證與站點身分不是網站本身，而是路由器側邏輯

### Port 5001
- 目前由 **Synology DSM** 提供內容
- 已有正確 Let’s Encrypt 憑證

---

## 四、核心問題

目前 `site2000.com` 的入口層是混雜的：

- `80` → 網站
- `443` → 路由器登入頁
- `5001` → DSM

這代表：
1. 主站 HTTPS 並沒有真正落到網站服務
2. 路由器目前攔截了對 `443` 的存取
3. 網站、路由器、NAS 之間沒有乾淨的入口分層

---

## 五、目前最合理的解釋

你現在的對外結構很可能是：

```text
site2000.com / www.site2000.com
        ↓
    同一個公網 IP
        ↓
 ┌───────────────┬─────────────────┬────────────────┐
 │ Port 80       │ Port 443        │ Port 5001      │
 │ nginx 網站    │ DrayTek 路由器  │ Synology DSM   │
 └───────────────┴─────────────────┴────────────────┘
```

所以你才會看到：
- HTTP 網站正常
- HTTPS 主站怪掉
- DSM 用 5001 才正常

---

## 六、風險與缺點

### 1. 主站 HTTPS 不成立
- 網站沒有真正用標準 443 正常對外
- 這對品牌、信任、瀏覽器安全性都不好

### 2. 路由器登入入口可能被公網直接碰到
- 若 `443` 導向 DrayTek 登入
- 代表你的網路設備管理面可能在外部可見

### 3. DSM 與網站沒有乾淨分流
- 現在是靠 port 硬分
- 長期維運會亂

---

## 七、建議的目標狀態

### 公開網站
- `https://www.site2000.com` → 網站（走 443）

### 路由器管理
- 不應直接讓公開網域 `443` 進去路由器登入頁
- 最好改為：
  - 僅內網可見
  - 或 VPN / Tailscale 後可見

### Synology 管理入口
- 建議未來改為：
  - `nas.site2000.com` 或 `admin.site2000.com`
- 並走獨立憑證與更清楚的入口邏輯

---

## 八、建議的下一步順序

### 第一優先
**把 443 從 DrayTek 路由器那邊收回來**

目標：
- 讓 `https://www.site2000.com` 真正回到網站服務

### 第二優先
**把路由器登入入口移出主網域的 443**

### 第三優先
**為 NAS / DSM 規劃獨立子網域**
- `nas.site2000.com`
- `admin.site2000.com`

---

## 結論

目前 `site2000.com` 的入口層不是壞掉，而是：

> **網站、路由器、DSM 三個角色擠在同一個公網入口裡，分工沒切乾淨。**

最重要的不是再加功能，而是先把入口邏輯收乾淨。
